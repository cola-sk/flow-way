# 绕川 Flow-Way

进京证摄像头绕行导航 App

## 项目结构

```
flow-way/
├── app/          # Flutter 客户端
├── server/       # Next.js API 服务端 (Vercel 部署)
└── README.md
```

## 快速开始

### 服务端

```bash
cd server
npm install
npm run dev    # 本地启动 http://localhost:3002
```

API 端点：
- `GET /api/cameras` — 获取所有摄像头坐标数据

### 客户端

```bash
cd app
flutter pub get
flutter run     # 在模拟器或真机运行
```

## 技术栈

- **客户端**: Flutter + flutter_map + 高德瓦片
- **服务端**: Next.js (App Router) + TypeScript
- **部署**: Vercel (服务端) + App Store / Play Store (客户端)
- **数据源**: jinjing365.com 摄像头坐标 (GCJ-02 坐标系)

- **坐标系与底层消除错位**：全屏导航页图层切换至高德 GCJ-02 瓦片图层（与路网及点位计算基准强一致），彻底修复了之前使用 OSM 导致的“数百米路网错位”。
- **车载级事件驱动定位刷新 (Event-Driven 定位)**：摒弃了倒计时的定时刷新，改为基于 `Geolocator` 的事件流 (`distanceFilter: 2` 米)。即：车辆移动超过 2 米即刻更新画面（顺滑无延迟），车辆静止（如等红灯）则暂停更新地图以防晕眩并省电。
- **直通导航**：用户在「Saved」已保存列表内，可点击线路右侧导航按钮，一键进入全屏导航模式并立即开始 GPS 实时循迹跟踪。
- 提供“偏航重发”和“前方摄像头预警”的语音播报能力 (TTS)。
- 整合了底层路径点持久化能力，导航结束后可通过快捷面板一键存入「Saved Route」。

### 7. 导航表单与交互 UI 对齐（参照大厂规范）
- **反转途径点 (Swap Waypoints)**：将“反转点位”按钮位置从底部提取到了途径点拖拽列表的右侧居中对齐，逻辑及布局完全看齐高德/苹果地图，提升业务操作直觉。

### 8. 数据一致性优化：“幽灵摄像头”与索引映射修复 (Index Mapping)
- **问题场景**：后台升级支持过滤被标记为“停用（Dismissed）”的摄像头，过滤操作缩减了原始摄像头数组长度，引发下标全体左移偏移。当算法返回路线碰线的 `hits` 时，前端基于全局数组读取就出现了严重错位（幽灵摄像头问题）。
- **解决方案**：在核心 `planAvoidCamerasRoute` 的过滤阶段引入 `indexMapping` 机制。即先构建新旧下标映射 `Record<newIdx, oldIdx>`，算法过程在过滤后的较短数组内高频执行（提高计算效率），在最终返回 `hits` 前，通过字典映射投影回源数据绝对下标 `originalHits = bestRoute.hits.map(idx => indexMapping[idx])`。从而保障了前后端摄像头池下标一致，彻底根治了假点预警。


## 核心算法揭秘：如何精准避开进京证摄像头

为了解决“完美避开所有抓拍点”这一复杂问题，系统在服务端实现了一套基于 **点线投影 + 矢量夹角判定 + 震荡退火寻优** 的智能探路算法。依靠几何数学与多次智能重组，最大程度压低“碰线”次数。

整个算法逻辑可以通俗地分为「如何判定被拍」和「如何规划绕开」两个层面：

### 1. 判定层面：我到底有没有被拍？

为了防止“草木皆兵”导致瞎绕路，新版算法的判定条件极其严苛且精准：

- **(1) 距离测算：点到线段的垂直投影（零死角测距）**
  此前的判定通常只测量“摄像头”到“导航路径每个坐标点”的距离，但这有一个致命缺陷：当在很长的直道上行驶时，坐标点可能相隔几百米，导致明明从摄像头下面开过去，程序的记录点却在数百米外。新算法引入了完整的几何向量投影（`distanceToSegment`），直接计算摄像头到“车辆行进轨迹线段”的垂直最短距离。因为精度提到了极限，**碰线判定红线被收紧到了仅 40 米**，彻底消除了隔壁平行车道或高架桥下摄像头的误杀。
  
- **(2) 视场角匹配：双向矢量解析与 40° 狭细视角**
  算完了距离还要算车头朝向。路面分布着大量类似“东西双向”的复合摄像头，系统引擎能准确将这类文字拆解为 `[90, 270]` 等多轴向矢量；同时，程序会抽离车辆途经这段线的实际航向角，只有当**车身轨迹与摄像头抓拍方向的夹角 < 40度**，才被判定为“正在被拍”。这使得路口大角度斜切弯、或反向车流能被智能忽略并安全放行。

### 2. 规划层面：怎样绕才能最聪明？

明确了哪些摄像头有威胁后，系统便开始拉黑禁区，通过“贪心搜索 + 模拟退火抖动”不断逼近全城最优解，每轮规划都在前一轮基础上演进，最多进行 20 轮。

- **(1) 布置微型地雷（贪心规避）**
  每次发起底层真实路网请求时（一次请求多条备选），算法会扫描新路线踩中了哪些雷。随后将这些雷连根拔起，并在坐标点上膨胀出一个 **约 35 x 35 米的微型多边形绝对禁区**，作为黑名单强制塞给下一次寻路请求。逼迫底层的道路规划引擎下次必须另辟蹊径（比如钻胡同、小路走）。
  
- **(2) 里程宽容度放大**
  只要新探出来的路，抓拍的摄像头数量**真的变少了**，即便系统让你绕大远路（最高允许绕跑到直线最佳里程的 **2.0 倍**），算法也会认定这是“壮士断腕”的全局最好路线保留下来。
  
- **(3) 死胡同智能破局（退火与随机震荡机制）**
  经常会遇到绝境：因为过河只有一条桥，或者所有主路全被封死，连续几轮怎么绕都会碰雷。为了防止卡在死局里，算法加入了类似“模拟退火”的随机破局动作：
  - **随机宽恕**：从黑名单里随机解冻 1~3 个无法避免的必经路口或桥头摄像头（承认避无可避，放开车道）。
  - **精准打击**：从当前拿到的“最好但也受阻”的路线上，随机挑出 1 颗漏网之监控强行拉黑（阻断退路，逼迫它去搜更隐秘的缝隙）。
  
  经过这样轻微的抖动与推翻重做，算法能像水流一样强行在“被探满的死胡同”中挤出一条最安全的路线。它使得原本极易假死的僵硬路网，变成了能在极复杂的十几个摄像头监控网中，耗时约 5 秒找出零/极少数摄像头的智能活地图。
