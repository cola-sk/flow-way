# 绕川 Flow-Way

进京证摄像头绕行导航 App

## 项目结构

```
flow-way/
├── app/          # Flutter 客户端
├── server/       # Next.js API 服务端 (Vercel 部署)
└── README.md
```

## 快速开始

### 服务端

```bash
cd server
npm install
npm run dev    # 本地启动 http://localhost:3002
```

API 端点：
- `GET /api/cameras` — 获取所有摄像头坐标数据

### 客户端

```bash
cd app
flutter pub get
flutter run     # 在模拟器或真机运行
```

## 技术栈

- **客户端**: Flutter + flutter_map + 高德瓦片
- **服务端**: Next.js (App Router) + TypeScript
- **部署**: Vercel (服务端) + App Store / Play Store (客户端)
- **数据源**: jinjing365.com 摄像头坐标 (GCJ-02 坐标系)

- 提供“偏航重现”和“靠近摄像头预警”的语音播报能力 (TTS)。
- 整合了底层路径点持久化能力，导航结束后可通过快捷面板一键存入「Saved Route」，记录本次导航的点位轨迹。

### 7. 数据一致性优化：“幽灵摄像头”与索引映射修复 (Index Mapping)
- **问题场景**：后台升级支持过滤被标记为“停用（Dismissed）”的摄像头，过滤操作缩减了原始摄像头数组长度，引发下标全体左移偏移。当算法返回路线碰线的 `hits` 时，前端基于全局数组读取就出现了严重错位（幽灵摄像头问题）。
- **解决方案**：在核心 `planAvoidCamerasRoute` 的过滤阶段引入 `indexMapping` 机制。即先构建新旧下标映射 `Record<newIdx, oldIdx>`，算法过程在过滤后的较短数组内高频执行（提高计算效率），在最终返回 `hits` 前，通过字典映射投影回源数据绝对下标 `originalHits = bestRoute.hits.map(idx => indexMapping[idx])`。从而保障了前后端摄像头池下标一致，彻底根治了假点预警。

## 导航避让算法（核心逻辑）

为解决进京证摄像头避让问题，系统实现了一套基于 **多次迭代 + 模拟退火（Jitter/Perturbation）** 的智能探路算法。该算法结合腾讯地图核心路网能力，专门针对“被摄像头包围”、容易陷入死胡同的路况进行了深度优化。在确保路线距离可控的条件下，最大程度压低碰线（经过摄像头）的次数。

### 算法核心策略（`planAvoidCamerasRoute`）

算法的演进包含以下步进与突变组合环节（最大支持 20 轮迭代）：

1. **多边形规避机制**
   - 依赖腾讯地图 API 的 `avoid_polygons` 属性（最大支持 32 个避让区）。
   - 将路径上碰撞到的摄像头实体，转换为边长约 35 米的微型避让区（通过经纬度 `±0.00035` 构造多边形）。
   
2. **结合绕路阈值的贪婪寻优**
   - 每轮请求均开启备选路线（`alternatives=true`）。
   - 将返回的各组路线与摄像头全量数据碰撞，评估“碰线数量”及“总里程”。
   - 若某轮探索到的新路线能在“里程不失控”（不超过当前最佳里程的 2.0 倍）前提下：
     - 指标提升（碰线更少，或者碰线一致但距离更短），则全局接纳此策略结果。
   - 正常阶段使用贪婪策略，将本轮路线新碰到的摄像头填入黑名单池（`avoid_polygons`），以便下回合强制系统绕开。

3. **进入局部最优的判定（死胡同探测）**
   - 若连续 2 轮内寻路结果碰线数量未能减少，或无法将“新摄像头”有效加入黑名单，则判定算法卡在“局部最优（如桥梁、必经路口等死胡同）”。
   
4. **模拟退火 / 随机抖动（Perturbation）破局拦截**
   - 触发条件：陷入局部最优。
   - 惩罚与宽恕混合动作：
     - **随机宽恕**：从已有拉黑的摄像头黑名单中，随机移除 1~3 个（可能是唯一出路的“必经点”）。
     - **精准打击**：从当前拿到的“最佳路线”碰到的摄像头中，随机挑选 1 个塞入黑名单（强制切断旧的最佳方案）。
   - 目的是在被卡死的路段强制撬开一条新缝隙，驱动下一轮腾讯 API 返回区别于此前的差异化路线。

5. **退出收敛条件**
   - 达成完美状态：碰到 0 个摄像头（直接 `break` 收工）。
   - 到达迭代阈值：超过 20 轮计算。最终向前端交付 `hits` 最小且符合里程边界的全局最佳路线。

> **优化效果总结**：相较早期容易陷入反复测算相似路线导致页面假死且无变化的缺陷，此退火重绕机制能用约 5~10 秒，在极复杂的“10+ 摄像头监控网”路况下找到仅剩 1~2 个必经过摄像头的收敛底线的最优解。
