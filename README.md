# 绕川 Flow-Way

进京证摄像头绕行导航 App

## 项目结构

```
flow-way/
├── app/          # Flutter 客户端
├── server/       # Next.js API 服务端 (Vercel 部署)
└── README.md
```

## 快速开始

### 服务端

```bash
cd server
npm install
npm run dev    # 本地启动 http://localhost:3002
```

API 端点：
- `GET /api/cameras` — 获取所有摄像头坐标数据

### 客户端

```bash
cd app
flutter pub get
flutter run     # 在模拟器或真机运行
```

## 技术栈

- **客户端**: Flutter + flutter_map + 高德瓦片
- **服务端**: Next.js (App Router) + TypeScript
- **部署**: Vercel (服务端) + App Store / Play Store (客户端)
- **数据源**: jinjing365.com 摄像头坐标 (GCJ-02 坐标系)

## 当前最核心问题（P0）

### 导航低效：疑似持续重复路径尝试
当前导航在部分场景下表现为“持续重复尝试相似路径”，体感为规划效率低、收敛慢，影响可用性与可信度。

### 现象描述
- 看起来一直在循环尝试路线，短时间内难以给出明显更优结果。
- 多段导航（起点-途径点-终点）时，整体耗时更明显。
- 用户感知为“导航在忙，但结果没有明显变化”。

### 影响
- 导航等待时间长，交互成本高。
- 用户对“避开摄像头”模式的稳定性预期下降。
- 在真实出行场景中，决策窗口被压缩。

### 下一步优化方向（建议优先级）
1. 降低重复尝试：对相似路径做去重与相似度阈值拦截，避免无效重算。
2. 设置硬性收敛条件：超过最大收益阈值后提前停止，而不是持续迭代。
3. 分层策略：先快速给出可用路线，再后台做有限次增强迭代。
4. 多段导航优化：针对每段设置独立上限，避免某一段无限拖慢整体。
5. 指标化观测：记录每轮改进幅度、重复率、平均收敛轮次，驱动参数调优。

> 结论：当前版本功能面已较完整，但“导航收敛效率”是最关键的产品体验瓶颈，应作为下一阶段首要治理目标。
