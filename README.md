# 绕川 Flow-Way

进京证摄像头绕行导航 App

## 项目结构

```
flow-way/
├── app/          # Flutter 客户端
├── server/       # Next.js API 服务端 (Vercel 部署)
└── README.md
```

## 快速开始

### 服务端

```bash
cd server
npm install
npm run dev    # 本地启动 http://localhost:3002
```

API 端点：
- `GET /api/cameras` — 获取所有摄像头坐标数据

### 客户端

```bash
cd app
flutter pub get
flutter run     # 在模拟器或真机运行
```

## 技术栈

- **客户端**: Flutter + flutter_map + 高德瓦片
- **服务端**: Next.js (App Router) + TypeScript
- **部署**: Vercel (服务端) + App Store / Play Store (客户端)
- **数据源**: jinjing365.com 摄像头坐标 (GCJ-02 坐标系)

- **坐标系与底层消除错位**：全屏导航页图层切换至高德 GCJ-02 瓦片图层（与路网及点位计算基准强一致），彻底修复了之前使用 OSM 导致的“数百米路网错位”。

- **直通导航**：用户在「Saved」已保存列表内，可点击线路右侧导航按钮，一键进入全屏导航模式并立即开始 GPS 实时循迹跟踪。

- 整合了底层路径点持久化能力，导航结束后可通过快捷面板一键存入「Saved Route」。

### 7. 导航表单与交互 UI 对齐（参照大厂规范）
- **反转途径点 (Swap Waypoints)**：将“反转点位”按钮位置从底部提取到了途径点拖拽列表的右侧居中对齐，逻辑及布局完全看齐高德/苹果地图，提升业务操作直觉。

### 8. 数据一致性优化：“幽灵摄像头”与索引映射修复 (Index Mapping)
- **问题场景**：后台升级支持过滤被标记为“停用（Dismissed）”的摄像头，过滤操作缩减了原始摄像头数组长度，引发下标全体左移偏移。当算法返回路线碰线的 `hits` 时，前端基于全局数组读取就出现了严重错位（幽灵摄像头问题）。
- **解决方案**：在路由 API 层（`/api/route/plan` 与 `/api/route/plan-step`）引入 `indexMapping` 机制。即先构建新旧下标映射 `Record<newIdx, oldIdx>`，算法过程在过滤后的较短数组内执行，在最终返回路线前通过映射还原为原始摄像头数组下标（`cameraIndicesOnRoute`）。从而保障了前后端摄像头池下标一致，彻底根治了假点预警。


## 核心算法揭秘：如何精准避开进京证摄像头

为了解决“完美避开所有抓拍点”这一复杂问题，系统在服务端实现了一套基于 **点线投影 + 矢量夹角判定 + 震荡退火寻优** 的智能探路算法。依靠几何数学与多次智能重组，最大程度压低“碰线”次数。

整个算法逻辑可以通俗地分为「如何判定被拍」和「如何规划绕开」两个层面：

### 1. 判定层面：我到底有没有被拍？

为了防止“草木皆兵”导致瞎绕路，新版算法的判定条件极其严苛且精准：

- **(1) 距离测算与精确投影：防路口侧向误杀与轨迹溯源**
  新版算法彻底弃用了简陋的距离阈值判定，引入了严格的**数学矢量投影算法 (`checkCameraOnSegment`)**：
  - **精准投影防误杀**：计算摄像头到车辆行进轨迹线段的垂直投影。当且仅当投影点真实落在线段内部（即车辆不仅离摄像头近，且必须在投影几何面上切实“越过”摄像头），且横向距离小于 40 米时，才判定为物理经过。彻底消除了由于“转弯路口距离近”而引发的误报（例如摄像头在路口外 30 米，车辆在路口转弯，直线距离虽近但车辆并未经过）。
  - **路口转向溯源与上下游几何判别 (Upstream/Downstream Geometry)**：针对放置在十字路口附近的难点摄像头，新增了严格的“拐弯溯源与上下游判别”逻辑。早期算法仅凭绝对距离（<40米）与来车方向来判定拐弯，导致了严重的漏报误伤（**Case记录：在测试“从隆昊肉类到家”的线路时，途经坐标 `116.649549, 39.901318` 的“自南向北”摄像头。由于该摄像头位于转弯点前方下游 30 多米处，旧算法粗暴地判定为“侧面拐入盲区”从而跳过匹配，导致应命中的摄像头被漏报**）。
  - 为了根治此类漏报，现已引入**方向向量点积（Dot Product）投影计算**：
    - 将“当前线段的行进向量”与“拐点指向摄像头的相对向量”进行内积计算，精准判定摄像头究竟在路口拐点的**上游（后方）**还是**下游（前方）**。
    - **若点积 < 0（上游）**：说明车辆从侧向拐入，且摄像头在车辆身后，确实拍不到，精准放行。
    - **若点积 ≥ 0（下游）**：说明车辆拐弯后，前方必然会直面并经过该摄像头，严格将其纳入碰线预警，杜绝漏报。
  
- **(2) 视场角匹配：双向矢量解析与兼容性方向容差**
  算完了距离还要算车头朝向。路面分布着大量类似“东西双向”的复合摄像头，系统引擎能准确将这类文字拆解为 `[90, 270]` 等多轴向矢量。针对对角线道路（如西北-东南走向）在地图系统中往往被粗略录入为“东西向”或“南北向”的普遍现象，引擎智能放宽了矢量方向的容忍度（**60度**），这搭配了前述的“矢量投影与路口溯源”双重几何屏障，既完美兼容了斜向道路上的方向命名模糊问题，又确保了绝不发生路口串流误报。

### 2. 规划层面：怎样绕才能最聪明？

明确了哪些摄像头有威胁后，系统便开始拉黑禁区，通过“贪心搜索 + 模拟退火抖动”不断逼近全城最优解，每轮规划都在前一轮基础上演进，最多进行 20 轮。

- **(1) 布置微型地雷（贪心规避）**
  每次发起底层真实路网请求时（一次请求多条备选），算法会扫描新路线踩中了哪些雷。随后将这些雷连根拔起，并在坐标点上膨胀出一个 **半径约 35 米（边长约 70 米量级）的微型多边形绝对禁区**，作为黑名单强制塞给下一次寻路请求。逼迫底层的道路规划引擎下次必须另辟蹊径（比如钻胡同、小路走）。
  
- **(2) 里程宽容度放大与严控离谱绕路 (Baseline Distance Anchoring)**
  只要新探出来的路，抓拍的摄像头数量**真的变少了**，系统允许进行一定程度的绕路。在早期版本中，如果地图底层 API 返回了一条为了少被拍而跨越几十公里的大绕行路线（例如从通州到通州却绕行几十公里到丰台区，长达 72km 的兜底备选），算法会由于盲目贪心而采纳该路线，并导致界面上**命中几十公里外毫不相关的摄像头**的荒谬现象。
  - **当前算法改进**：现在算法会在第一轮探索时提取最直接的原始路线长度作为**基准里程 (`baselineDist`)**。在随后的多轮多边形阻断探索中，一旦发现某条备选路线的总长度超过了基准里程的 **2.0 倍** 或是比基准 **多出 10 公里**，算法会毫不犹豫地将其丢弃。这就保证了在“努力绕行避坑”和“拒绝离谱大绕远”之间取得完美的平衡。
  
- **(3) 死胡同智能破局（退火与随机震荡机制）**
  经常会遇到绝境：因为过河只有一条桥，或者所有主路全被封死，连续几轮怎么绕都会碰雷。为了防止卡在死局里，算法加入了类似“模拟退火”的随机破局动作：
  - **随机宽恕**：从黑名单里随机解冻 1~3 个无法避免的必经路口或桥头摄像头（承认避无可避，放开车道）。
  - **精准打击**：从当前拿到的“最好但也受阻”的路线上，随机挑出 1 颗漏网之监控强行拉黑（阻断退路，逼迫它去搜更隐秘的缝隙）。
  
  经过这样轻微的抖动与推翻重做，算法能像水流一样强行在"被探满的死胡同"中挤出一条最安全的路线。它使得原本极易假死的僵硬路网，变成了能在极复杂的十几个摄像头监控网中，耗时约 5 秒找出零/极少数摄像头的智能活地图。

- **(4) 人工添加途径点（Guided Helper）旁路推演**：
  - 在每次产生局部死胡同（`noImprovementCount >= 2`）且即将执行破坏性的算法退火/抖动打乱之前，触发独立推演。
  - 算法会生成一个或多个辅助绕行途经点，将其封装在这个特殊的推演 `tryGuidedHelperRoute` 里面向腾讯 API 单独发一次验证请求。
  - 它的引入不再会破坏原来好的路线：拿到新路线后会跟现有的最好的路线（基准线）严格比拼代价——仅当由于途径点产生的新路线不超过程规绕路距离上限（`baseline * 2.0` 等）且**撞的摄像头数量真真实实变少了或者里程变短了**，才会用它全面替换主路线，并据此继续收集摄像头黑名单。否则静默丢弃，算法继续走常规退火机制。

- **(5) "换条路试试" - 多策略走廊排除（Retry with Corridor Exclusion）**
  当算法收敛后仍有无法绕开的摄像头时，用户可以点击 **"换条路试试"** 按钮触发第二轮搜索。该机制的核心原理：
  - **走廊排除**：将上一次规划路线均匀采样 6 个关键节点，以约 80 米半径膨胀为禁行多边形，强制腾讯 API 在完全不同的道路走廊中搜索新路线。
  - **方向随机扰动**：在前 4 轮搜索中，自动注入路线中段的随机偏移 waypoint（垂直于起终点连线方向偏移 500-1500 米），打破地图引擎对默认最优路线的固有偏好。
  - **累积排除**：每次重试都会累积之前所有已探索路线作为排除对象，避免反复搜索相同走廊。支持连续多次重试。

### 3. 当前 plan-step 接口行为说明

- `plan-step` 目前采用"单次请求内完成完整避让搜索"的模式：服务端一次性跑完避让算法并返回 `done: true`。
- 支持 `excludePolylines` 参数：传入之前规划过的路线折线坐标数组，实现"换条路试试"的走廊排除功能。
- 前端虽然保留了逐轮状态展示 UI，但在当前模式下通常会在首个响应即收敛。
- 避让失败时会回退到直线兜底，并返回该兜底路径上的 `cameraIndicesOnRoute` 以支持前端红色标记与提示。
